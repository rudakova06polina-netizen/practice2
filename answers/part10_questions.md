## Вопрос 1

Что выведет этот код? Объясните порядок выполнения блоков.

```java
class Demo {
    static { System.out.print("A "); }
    { System.out.print("B "); }
    Demo() { System.out.print("C "); }
    public static void main(String[] args) {
        System.out.print("D ");
        new Demo();
        System.out.print("E ");
        new Demo();
    }
}
```

**Предсказанный вывод:**

```
A D B C E B C
```

**Объяснение:** При выполнении этого кода первым делом при загрузке класса Demo в память срабатывает статический блок инициализации, который выводит A. Затем управление переходит в метод main, где первой инструкцией выводится D. После этого создаётся первый объект new Demo(). Перед вызовом конструктора для каждого нового объекта выполняется нестатический блок инициализации, выводящий B, а затем уже сам конструктор выводит C. Далее в методе main выводится E и создаётся второй объект new Demo(). При его создании снова сначала отрабатывает нестатический блок с выводом B, а потом конструктор с выводом C.

## Вопрос 2

Класс `Child` в пакете `b`, класс `Parent` в пакете `a`. `Child extends Parent`. Какие члены `Parent` доступны в `Child`? А какие доступны в другом классе пакета `b`, который НЕ наследует `Parent`?

**Ваш ответ:**
В классе Child (наследует Parent из другого пакета) доступны public и protected члены.
В другом классе пакета b (без наследования) доступны только public члены Parent.

## Вопрос 3

Можно ли создать sealed-класс, у которого все наследники — record? Какое преимущество это даёт при использовании в switch?

**Ваш ответ:** Да, можно. Когда все наследники sealed-класса или интерфейса являются record, компилятор получает полную информацию о структуре каждого варианта — не только о типе, но и о компонентах. Это позволяет использовать паттерн деструктуризации в switch, автоматически извлекая поля

## Вопрос 4

Что выведет этот код? Объясните, что такое String Pool и как работает `intern()`.

```java
String a = "Hello";
String b = new String("Hello");
String c = b.intern();
System.out.println(a == b);
System.out.println(a == c);
System.out.println(b == c);
```

**Предсказанный вывод:**

```
false
true
false
```

**Объяснение:** String Pool — это специальная область памяти в куче, где JVM хранит уникальные экземпляры строковых литералов. Когда вы создаёте строку через двойные кавычки, JVM сначала проверяет, есть ли уже такая строка в пуле. Если есть — возвращает ссылку на неё, если нет — добавляет новую строку в пул.
Как работает intern():
Метод intern() проверяет, содержится ли строка в пуле. Если да — возвращает ссылку на строку из пула. Если нет — добавляет текущую строку в пул и возвращает ссылку на неё.

## Вопрос 5

Почему этот код не компилируется? Как исправить?

```java
interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B { }
```

**Ваш ответ:** Этот код не компилируется из-за конфликта имён. Класс C наследует два одинаковых метода hello() от интерфейсов A и B
```java
class C implements A, B {
    @Override
    public void hello() {
        A.super.hello(); 
    }
}
```

## Вопрос 6

В чём разница между `StringBuilder` и `StringBuffer`? Приведите конкретный сценарий, когда нужен именно `StringBuffer`.

**Ваш ответ:**
StringBuffer является потокобезопасным — все его публичные методы синхронизированы, что позволяет безопасно использовать один объект StringBuffer в нескольких потоках одновременно. Однако эта синхронизация создаёт дополнительные накладные расходы, поэтому StringBuffer работает медленнее. StringBuilder же не синхронизирован, он не гарантирует потокобезопасность, но зато работает быстрее, так как не тратит время на блокировки.
Конкретный сценарий: веб-сервер, где несколько потоков одновременно обрабатывают запросы и пишут информацию в общий буфер лога.

## Вопрос 7

Что выведет этот код? Объясните, что происходит с переменной `x` в лямбде.

```java
int x = 10;
Runnable r = () -> System.out.println(x);
// x = 20;  // что будет, если раскомментировать?
r.run();
```

**Предсказанный вывод:**

```
10
```

**Объяснение:**
В текущем виде код выведет 10, потому что лямбда захватывает переменную x и использует её значение на момент выполнения.
Однако если раскомментировать строку x = 20, программа не скомпилируется. Причина в том, что в Java лямбда-выражения могут захватывать только те локальные переменные, которые являются effectively final — то есть не изменяются после инициализации. Изначально переменная x этому правилу удовлетворяет, но попытка присвоить ей новое значение 20 нарушает это условие, и компилятор выдаст ошибку.

**Ваш пример и объяснение:**
```java
x -> x > 0
```
Эту лямбду нельзя заменить на ссылку на метод. Лямбда выполняет операцию сравнения с нулём — это не вызов одного существующего метода, а логическое выражение. Нет ни одного стандартного метода, который принимает число и возвращает true, если оно больше нуля..

## Вопрос 9

Чем `EnumMap` быстрее `HashMap` для ключей-enum? Опишите внутреннюю реализацию в одном предложении.

**Ваш ответ:** EnumMap быстрее HashMap, потому что вместо хэш-кодов и разрешения коллизий он хранит данные в компактном массиве, где индекс элемента вычисляется напрямую из порядкового номера ключа (ordinal()), что даёт доступ за O(1) без вычисления хэша и без работы со связными списками или красно-чёрными деревьями.

## Вопрос 10

Дан `record Point(int x, int y) {}`. Что сгенерирует компилятор автоматически? Можно ли добавить в record изменяемое поле? Можно ли наследоваться от record?

**Ваш ответ:** Компилятор сгенерирует для этого record конструктор, методы доступа x() и y(), а также реализации equals(), hashCode() и toString(). Все поля будут объявлены как private final, что делает record неизменяемым. Добавить в record обычное изменяемое поле невозможно, исключение статтические поля. Наследоваться от record тоже нельзя: компилятор делает все record-классы неявно final.  Более того, сам record не может наследовать другой класс, однако record может реализовывать любое количество интерфейсов
